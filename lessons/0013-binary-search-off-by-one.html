BS: El Off-by-One mid +/- 1

La fuente mas comun de bugs en Binary Search no es la logica — es decidir si el mid actual puede ser la respuesta o no.

1. El Problema

Cuando haces BS con while left < right, en cada paso descartas una porcion del array. La pregunta es: el mid actual, puede ser la respuesta o no?

Si te equivocas obtienes:

  • right = mid - 1 cuando debio ser right = mid → descartas la respuesta
  • right = mid cuando debio ser right = mid - 1 → loop infinito
El off-by-one no es "siempre inevitable". No es un error aleatorio. Es una decision consciente que tomas basada en una pregunta: "mid es candidato valido o no?"

2. El Loop Invariant

Que contiene [left..right]?

Cuando usas while left < right, el invariante es:
"La respuesta esta en [left..right], si es que existe."

Cada iteracion debe reducir el rango pero sin perder la respuesta.

Esto significa: cuando examinas mid, preguntas:

  • Puedo probar que mid NO es la respuesta? → descartalo con +1 o -1
  • No puedo descartarlo? → conservalo con = mid

3. Los 4 Movimientos

Movimiento Significa Riesgo
left = mid + 1 Descarto mid y todo lo que esta a su izquierda Seguro si mid no puede ser la respuesta
right = mid - 1 Descarto mid y todo lo que esta a su derecha Seguro si mid no puede ser la respuesta
right = mid Conservo mid como candidato, descarto lo que esta a su derecha Seguro si mid podria ser la respuesta
left = mid Conservo mid como candidato, descarto lo que esta a su izquierda PELIGROSO: puede causar loop infinito si left + 1 == right
Regla de oro: El movimiento left = mid es el unico que puede causar loop infinito (cuando solo quedan 2 elementos). Si lo usas, necesitas el ceiling en mid: mid = left + (right - left + 1) / 2. Los otros 3 movimientos son siempre seguros.

4. Visualizacion: Find Minimum

Veamos nums = [3,4,5,6,7,1,2], buscando el minimo.

Decisión: Cuando nums[mid] < first_half_init, mid esta en la segunda mitad y podria ser el minimo. No podemos descartarlo → right = mid.
Iteracion 1: left=0, right=6, mid=3
L 3 4 5 6 7 1 R

nums[3]=6 >= first_half=3 → primera mitad, no puede ser el minimo → left = mid + 1 = 4

Iteracion 2: left=4, right=6, mid=5
L 7 1 2 R

nums[5]=1 < first_half=3 → segunda mitad, podria ser el minimo → right = mid = 5

Si usaras right = mid - 1 = 4: descartarias el 1! left=4, right=4 → nums[4]=7. Respuesta incorrecta.

Iteracion 3: left=4, right=5, mid=4
L 7 1 R

nums[4]=7 >= 3 → primera mitad → left = mid + 1 = 5

Fin: left=5, right=5
L R 1

nums[5] = 1 → respuesta correcta!

5. Comparacion: Tu Caso vs Templetes

Template Condicion mid puede ser la respuesta? Accion
Find Min (tu version) nums[mid] < first_half Si right = mid
Find Min (canonica) nums[mid] < nums[right] Si right = mid
BS clasico nums[mid] > target No (ya lo revisamos) right = mid - 1
Search Rotated nums[mid] != target No (si fuera, ya retornamos) mid +/- 1
BS en respuesta (min factible) feasible?(mid) Si right = mid
Find Peak nums[mid] < nums[mid+1] No (el pico esta a la derecha) left = mid + 1
Patron: Cada vez que la condicion te dice "esto podria ser la respuesta", usas = mid (sin +/- 1). Cada vez que la condicion descarta mid, usas mid +/- 1.

6. El Unico Caso Peligroso: left = mid

right = mid es siempre seguro. Pero left = mid puede causar loop infinito. Por que?

left = mid SIN ceiling:

left=3, right=4, mid = 3 + (4-3)/2 = 3

L X R

left = mid = 3 → left=3, right=4 → no avanzamos! 🔁

left = mid CON ceiling:

left=3, right=4, mid = 3 + (4-3+1)/2 = 4

L R X

left = mid = 4 → left=4, right=4 → loop termina. Seguro!

Nunca uses left = mid sin el ceiling. Es la unica combinacion que puede causar loop infinito. right = mid, left = mid + 1 y right = mid - 1 son siempre seguros (siempre reducen el rango).

7. Ponte a Prueba

Pregunta 1

En Find Min, nums[mid] = 2, first_half = 5. mid puede ser el minimo?

Ver respuesta
Si. 2 < 5, asi que mid esta en la segunda mitad (la de los numeros chicos). Podria ser el minimo → right = mid.

Pregunta 2

En Find Min, nums[mid] = 8, first_half = 3. mid puede ser el minimo?

Ver respuesta
No. 8 >= 3, asi que mid esta en la primera mitad (los numeros grandes). El minimo no puede estar aqui → left = mid + 1.

Pregunta 3

En BS clasico, nums[mid] = 15, target = 10. Usas right = mid o right = mid - 1?

Ver respuesta
right = mid - 1. Por que? Porque ya verificaste que nums[mid] != target (si fuera igual, ya habrias retornado). Como 15 > 10, sabes que mid NO es la respuesta y todo lo que esta a su derecha tampoco. Descartalo completamente.

Pregunta 4

Que pasa si cambias right = mid por right = mid - 1 en Find Min con [4,5,6,7,8,1,2,3]?

Ver respuesta
El minimo es 1 en indice 5.
Cuando mid=5, nums[5]=1 y haces right=4, descartas el indice 5 y el minimo se pierde. El loop termina con nums[4]=8 como respuesta → incorrecto.

Pregunta 5

Por que left = mid puede causar loop infinito pero right = mid no?

Ver respuesta
Porque mid se calcula con piso (truncado hacia abajo). Si left=3, right=4, mid = 3 + (4-3)/2 = 3. left = mid = 3 → no avanza. 🔁

Pero con right = mid: mid=3, right=3 → left=3, right=3 → termina. O left = mid + 1 = 4 → left=4, right=4 → termina.

El piso en mid hace que left=mid sea peligroso, pero right=mid siempre reduce el rango.

8. Resumen: Arbol de Decision

Al examinar mid:

if nums[mid] es la respuesta?
  return mid

if puedo probar que mid NO es la respuesta?
  left  = mid + 1   # o
  right = mid - 1

else  # mid PODRIA ser la respuesta
  if left = mid?   # PELIGRO: necesita ceiling en mid
  if right = mid?  # Seguro siempre
