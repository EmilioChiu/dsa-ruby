Lección 0004: LRU Cache - La Sinergia Perfecta

El Problema: Imagina que tienes una memoria muy rápida pero muy pequeña (Capacidad K). Cuando se llena y quieres guardar algo nuevo, ¿a quién sacas?

La política LRU (Least Recently Used) dice: "Saca al que no hemos usado en más tiempo".

1. ¿Por qué es difícil?

Para una entrevista de nivel Top, te pedirán que las operaciones get(key) y put(key, value) sean O(1).

  • ¿Un Array? get es O(n) si no está ordenado. Si movemos elementos, es O(n). FAIL.
  • ¿Un Hash Map solo? get es O(1), pero ¿cómo sabemos cuál es el más viejo? No hay orden. FAIL.
  • ¿Una Linked List sola? Mantiene el orden, pero buscar una llave es O(n). FAIL.

2. La Solución: El Híbrido

Combinamos un Hash Map y una Doubly Linked List (DLL).

Hash Map: { llave => Nodo en la DLL }

↓

DLL: [Cabeza: MRU] ⇄ [Nodo] ⇄ [Nodo] ⇄ [Cola: LRU]

MRU: Most Recently Used | LRU: Least Recently Used
La lógica de la sinergia:
  1. El Hash Map nos da acceso instantáneo O(1) a cualquier nodo en la lista.
  2. La DLL nos permite "desenganchar" un nodo y moverlo al principio (Cabeza) en O(1).
  3. Al insertar uno nuevo, si excedemos capacidad, eliminamos el nodo que está en la Cola (el más viejo).

3. Anatomía de la Implementación

# Estructura del Nodo para la DLL
class Node
  attr_accessor :key, :value, :prev, :next
  def initialize(key, value)
    @key = key
    @value = value
  end
end

# La Estrategia en pseudocódigo
def get(key)
  return -1 unless map.has?(key)
  node = map[key]
  move_to_head(node) # ¡Es el más nuevo ahora!
  return node.value
end

def put(key, value)
  if map.has?(key)
    update_and_move_to_head(node)
  else
    evict_if_full
    add_new_node_to_head
  end
end

4. ¿Por qué Doubly Linked List?

Si fuera una Singly Linked List, para borrar un nodo necesitaríamos el puntero al anterior, lo cual nos obligaría a recorrer la lista (O(n)). Al ser Doubly, cada nodo ya sabe quién es su anterior. Borrar es O(1).
