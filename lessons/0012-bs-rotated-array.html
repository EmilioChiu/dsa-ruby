BS en Rotated Array O(log n)

El array esta ordenado... pero "roto". Como aplicas Binary Search cuando la posicion del pivot es desconocida? La clave: al menos una mitad siempre esta ordenada.

1. Que es un Array Rotado?

Un array ordenado ascendentemente se rota cuando tomamos algunos elementos del inicio y los movemos al final. Por ejemplo:

Original: [0, 1, 2, 4, 5, 6, 7]

0 1 2 4 5 6 7

Rotado en k=3: [4, 5, 6, 7, 0, 1, 2]

4 5 6 7 0 1 2

Los elementos rosados se movieron al inicio. Los verdes quedaron en su orden original.

Propiedad fundamental: En cualquier array rotado, al menos una mitad (izquierda o derecha) esta completamente ordenada. Esto es lo que explota Binary Search.

El pivot es el punto donde ocurrio la rotacion. En el ejemplo, el pivot esta entre el 7 y el 0. Los elementos a la izquierda del pivot son mayores que los de la derecha.

2. Search in Rotated Sorted Array (LC 33)

Problema: Dado un array rotado nums y un target, encuentra el indice del target o retorna -1.

La Intuicion

En cada paso, calculamos mid. Luego determinamos cual mitad esta ordenada comparando nums[left] con nums[mid]:

  • Si nums[left] <= nums[mid] → la mitad izquierda esta ordenada
  • Sino → la mitad derecha esta ordenada

Una vez que sabemos cual mitad esta ordenada, preguntamos: "El target esta dentro de esa mitad ordenada?":

  • Si si → buscas ahi
  • Si no → buscas en la otra mitad

La Plantilla

def search(nums, target)
  left  = 0
  right = nums.length - 1

  while left <= right
    mid = left + (right - left) / 2
    return mid if nums[mid] == target

    if nums[left] <= nums[mid]         # mitad izquierda ordenada
      if nums[left] <= target && target < nums[mid]
        right = mid - 1                   # target en la izquierda
      else
        left  = mid + 1                   # target en la derecha
      end
    else                                   # mitad derecha ordenada
      if nums[mid] < target && target <= nums[right]
        left  = mid + 1                   # target en la derecha
      else
        right = mid - 1                   # target en la izquierda
      end
    end
  end

  -1
end

Visualizacion Paso a Paso

Buscamos target = 0 en nums = [4, 5, 6, 7, 0, 1, 2]:

Paso 1: left=0, right=6, mid=3
L 4 5 6 7 0 1 R

nums[left]=4 <= nums[mid]=7 → izquierda ordenada [4,5,6,7]. target=0 NO esta entre 4 y 7 → vamos a la derecha.

left = mid + 1 = 4

Paso 2: left=4, right=6, mid=5
L 0 1 R

nums[left]=0 <= nums[mid]=1 → izquierda ordenada [0,1]. target=0 SI esta entre 0 y 1 → vamos a la izquierda.

right = mid - 1 = 4

Paso 3: left=4, right=4, mid=4
L 0 R

nums[4] == 0 == target → encontrado! Retorna 4.

Detalle sutil: Comparamos con nums[left] <= nums[mid] (no <). Por que? Porque si left == mid (array de 2 elementos), necesitamos que la condicion sea true para considerar la izquierda como ordenada.

3. Find Minimum in Rotated Array (LC 153)

Problema: Encuentra el elemento minimo en un array rotado. Todos los elementos son unicos.

La Intuicion

Aqui no buscamos un target — buscamos el pivot. La clave es comparar nums[mid] con nums[right]:

  • Si nums[mid] > nums[right] → el pivot esta en la derecha (porque lo normal seria que mid < right, asi que hay una "anomalia")
  • Si nums[mid] <= nums[right] → el pivot esta en la izquierda (incluyendo mid)
def find_min(nums)
  left  = 0
  right = nums.length - 1

  while left < right
    mid = left + (right - left) / 2

    if nums[mid] > nums[right]
      left  = mid + 1     # minimo esta en la derecha
    else
      right = mid            # minimo esta en la izquierda (incl. mid)
    end
  end

  nums[left]
end

Visualizacion

nums = [4, 5, 6, 7, 0, 1, 2]:

Paso 1: left=0, right=6, mid=3
L 4 5 6 7 0 1 R

nums[mid]=7 > nums[right]=2 → minimo a la derecha. left = mid + 1 = 4

Paso 2: left=4, right=6, mid=5
L 0 1 R

nums[mid]=1 <= nums[right]=2 → minimo a la izquierda (incl. mid). right = mid = 5

Paso 3: left=4, right=5, mid=4
L 0 R

nums[mid]=0 <= nums[right]=1 → minimo a la izquierda. right = mid = 4

Paso 4: left=4, right=4. left == right → retorna nums[4] = 0.
Nota: La condicion while left < right (no <=) porque cuando left == right, hemos encontrado el minimo. Este template es el mismo que usaste en BS en respuesta (leccion 0011).

4. Find Peak Element (LC 162)

Problema: Un elemento pico es estrictamente mayor que sus vecinos. Encuentra cualquier pico en O(log n).

Aunque el array NO esta ordenado (ni rotado), el patron de BS aplica porque podemos descartar mitades basados en una propiedad local:

  • Si nums[mid] < nums[mid + 1] → el pico esta a la derecha (la pendiente sube, asi que eventualmente debe bajar)
  • Si nums[mid] >= nums[mid + 1] → el pico esta a la izquierda (incluyendo mid, la pendiente baja)
def find_peak_element(nums)
  left  = 0
  right = nums.length - 1

  while left < right
    mid = left + (right - left) / 2

    if nums[mid] < nums[mid + 1]
      left  = mid + 1     # pico a la derecha
    else
      right = mid            # pico a la izquierda (incl. mid)
    end
  end

  left                     # cualquier indice es un pico
end
Cuidado: En LC 162, el problema garantiza que nums[-1] = nums[n] = -infinito. Esto asegura que siempre existe al menos un pico y que la estrategia de "seguir la pendiente" funciona.

5. Comparacion de los 3 Patrones

Search (LC 33) Find Min (LC 153) Peak (LC 162)
Condicion de comparacion nums[left] vs nums[mid] nums[mid] vs nums[right] nums[mid] vs nums[mid+1]
Que buscas? Un valor exacto El minimo (pivot) Cualquier pico local
Loop condition left <= right left < right left < right
Update left mid + 1 mid + 1 mid + 1
Update right mid - 1 mid mid
Retorno mid o -1 nums[left] left

6. Ponte a Prueba

Pregunta 1

En nums = [3, 4, 5, 1, 2], cual es el array original antes de rotar?

Ver respuesta
[1, 2, 3, 4, 5] — rotado en k=3.

Pregunta 2

En LC 33, que pasa si nums[left] <= nums[mid] es true y el target NO esta en ese rango? A donde vas?

Ver respuesta
Vas a la derecha (left = mid + 1). Como la mitad izquierda esta ordenada y el target no esta ahi, la unica posibilidad es la mitad derecha (aunque este "rota").

Pregunta 3

En LC 153, por que usamos while left < right en vez de while left <= right?

Ver respuesta
Porque el minimo es el unico elemento que queda cuando left == right. Con <=, si left == right, calcularias mid y podrias comparar nums[mid] consigo mismo, causando un loop infinito o un resultado incorrecto. El template de "estrechar rango" usa <.

Pregunta 4

En Find Peak Element, por que nums[mid] < nums[mid + 1] implica que el pico esta a la derecha?

Ver respuesta
Porque los extremos del array son -infinito. En mid, la pendiente sube (nums[mid] < nums[mid+1]). Como eventualmente el array debe bajar hacia -infinito en el extremo derecho, en algun punto la tendencia se invierte y ahi hay un pico. No puede ser que la subida sea infinita.
